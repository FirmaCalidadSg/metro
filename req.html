Especificación de Requerimientos de Software

Sistema de Capacidades Team (SICATEAM)

Propósito del Software: Desarrollar un sistema que permita la consulta en línea sobre la gestión del uso de capacidades instaladas de las plantas, identificando causas de pérdidas en los equipos, porcentaje de efectividad perdido, disminución de capacidad de producción y registro de piezas no conformes, mediante el registro de información de producción y tiempos perdidos por turno.

1. Requerimientos Funcionales

1.0 Gestión de Usuarios:

RF1.0.1: El sistema debe proporcionar una Autenticación segura para el acceso

RF1.0.2: El sistema debe proporcionar roles para el acceso

  • operador
  • supervisor
  • administrador

1.1 Gestión de Menú Principal

RF1.1: El sistema debe proporcionar una barra de navegación principal con las siguientes opciones:

  • Nuevo
  • Consultas
  • Reportes
  • Administración

1.2 Módulo de Administración

RF2.1: El sistema debe incluir un menú administrativo con las siguientes opciones:

  • Profile (Gestión de perfiles de usuario)
  • Definiciones (Configuraciones generales)
  • Países (Catálogo de países)
  • Ciudades (Catálogo de ciudades)
  • Equipos (Gestión de equipamiento)
  • Daños de Equipos (Registro y seguimiento de fallas)
  • Planta (Gestión de plantas productivas)
  • Proceso (Configuración de procesos)
  • Línea (Gestión de líneas de producción)
  • Producto (Catálogo de productos)
  • Línea y Producto (Relación entre líneas y productos)
  • Paro (Registro de paros)
  • SubParo (Categorización de paros)
  • Turno (Gestión de turnos)
  • Todos los documentos (Gestión documental)

1.3 Gestión de Plantas y Capacidades

  • RF3.1: El sistema debe permitir registrar y mantener un catálogo de plantas productivas
  • RF3.2: El sistema debe permitir registrar la capacidad instalada por planta
  • RF3.3: El sistema debe permitir consultar en tiempo real las capacidades disponibles
  • RF3.4: El sistema debe mantener un histórico de capacidades utilizadas

1.4 Monitoreo de Equipos

  • RF4.1: El sistema debe registrar y monitorear el estado de los equipos en tiempo real
  • RF4.2: El sistema debe identificar y registrar fallas en los equipos
  • RF4.3: El sistema debe calcular el porcentaje de efectividad perdida por fallas
  • RF4.4: El sistema debe generar alertas automáticas cuando se detecten fallas

1.5 Gestión de Producción

  • RF5.1: El sistema debe registrar la producción por turno
  • RF5.2: El sistema debe contabilizar las piezas conformes producidas
  • RF5.3: El sistema debe registrar las piezas no conformes y sus causas
  • RF5.4: El sistema debe calcular indicadores clave:
    • Eficiencia de producción por turno
    • TEEP (Total Effective Equipment Performance)
    • MTTR (Mean Time To Repair)

2. Requerimientos No Funcionales

2.1 Usabilidad

  • RNF1.1: Interface web responsive e intuitiva
  • RNF1.2: Menú de navegación claro y organizado jerárquicamente
  • RNF1.3: Tiempo de aprendizaje del sistema por usuario nuevo menor a 4 horas
  • RNF1.4: Documentación en línea y ayuda contextual

2.2 Rendimiento

  • RNF2.1: Tiempo de respuesta máximo de 3 segundos para consultas
  • RNF2.2: Capacidad de procesamiento de al menos 1000 transacciones por minuto
  • RNF2.3: Disponibilidad del sistema 24/7 con uptime de 99.9%

2.3 Seguridad

  • RNF3.1: Autenticación de usuarios mediante credenciales únicas
  • RNF3.2: Registro de todas las operaciones (audit trail)
  • RNF3.3: Cifrado de datos sensibles
  • RNF3.4: Backup automático cada 24 horas

2.4 Escalabilidad

  • RNF4.1: Arquitectura modular que permita crecimiento
  • RNF4.2: Capacidad de integración con otros sistemas empresariales
  • RNF4.3: Soporte para múltiples plantas y ubicaciones

3. Requerimientos de Interfaz

3.1 Interfaces de Usuario

  • RI1.1: Dashboard principal con KPIs en tiempo real
  • RI1.2: Formularios de registro de información
  • RI1.3: Pantallas de monitoreo de equipos
  • RI1.4: Interfaz de generación de reportes
  • RI1.5: Menú de navegación principal con iconos intuitivos
  • RI1.6: Submenús desplegables para opciones administrativas

3.2 Interfaces de Sistema

  • RI2.2: Conexión con sistemas de control de planta
  • RI2.3: Integración con bases de datos existentes

4. Requerimientos de Datos

4.1 Almacenamiento

  • RD1.1: Base de datos relacional para datos transaccionales
  • RD1.2: Sistema de almacenamiento para históricos
  • RD1.3: Respaldo periódico de información

4.2 Integridad

  • RD2.1: Validación de datos en tiempo real
  • RD2.2: Consistencia en la información entre módulos
  • RD2.3: Trazabilidad de cambios en los datos

5. Restricciones Técnicas

  • RT1: Desarrollo en PHP y JavaScript
  • RT2: Base de datos SQLSERVER22/MySQL/MariaDB
  • RT3: Arquitectura web responsive
  • RT4: Compatibilidad con navegadores modernos
